# 自动游玩与平衡测试

《朕不南渡》要能长期游玩，就必须持续自动跑局。目标不是让机器找“最优解”，而是及早发现三类问题：

- 某条路线数值飞天，玩家无脑重复即可通关。
- 某个危机过早崩盘，玩家还没理解系统就被历史碾死。
- 某些系统永远不起作用，只有文案存在感。

## 测试目标

自动游玩要回答这些问题：

1. 玩家是否总在前 3 回合财政崩盘？
2. 守城路线、议和拖延路线、清算筹饷路线、勤王路线是否都能跑出有效结果？
3. 任用李纲、清查军饷、开内帑、假议和等强动作是否过强？
4. 主和压力、金军威压、粮价、禁军欠饷是否有合理波动，而不是一路单调爆炸？
5. 派系反扑是否有存在感，但不至于把所有改革都吞掉？
6. 玩家失败时是否有可复盘原因？

## 自动玩家类型

至少做 8 类策略代理，每类跑固定种子和随机种子。

| 自动玩家 | 策略 |
| --- | --- |
| 守城派 | 优先李纲、修城、备器、排查城门 |
| 清算派 | 优先密令查账、殿前对质、抄没追赃 |
| 议和拖延派 | 优先外交拖延、暗中修城和催勤王 |
| 勤王派 | 优先发诏、补饷、调西军、保河东 |
| 民生派 | 优先粮价、开仓、安民、减轻征发 |
| 铁腕派 | 高强度罢免、下狱、戒严、军纪 |
| 平衡派 | 每回合按最大危机分配行动 |
| 随机派 | 合法动作随机选择，用来找边界崩坏 |

每个自动玩家都要记录：

- 每回合选择了哪些事件。
- 召见了谁。
- 下了哪些诏令。
- 花了多少钱粮。
- 哪些事项推进或恶化。
- 最终结局和关键失败原因。

## 路线覆盖矩阵

每次平衡测试至少覆盖：

| 维度 | 取值 |
| --- | --- |
| 开局压力 | 标准 / 高金军 / 高财政亏空 / 高派系反扑 |
| 玩家路线 | 守城 / 清算 / 议和 / 勤王 / 民生 / 铁腕 / 平衡 |
| LLM 模式 | 真实 LLM / 固定模拟器 / 低成本摘要模式 |
| 随机种子 | 10 个固定种子 + 20 个随机种子 |

MVP 期每次版本至少跑：

```text
8 个自动玩家 × 10 个固定种子 = 80 局
```

内容扩展期至少跑：

```text
8 个自动玩家 × 30 个种子 = 240 局
```

## 核心指标

### 结局分布

| 指标 | 健康区间 |
| --- | --- |
| 前 5 回合失败率 | < 15% |
| 标准难度通关率 | 25%-55% |
| 屈辱和议率 | 15%-35% |
| 城破率 | 20%-45% |
| 中兴大胜率 | < 15% |

健康区间不是绝对真理，主要用来发现异常。

### 数值轨迹

每局记录：

- 国库。
- 内帑。
- 京城粮。
- 民心。
- 君威。
- 汴京守备。
- 金军威压。
- 勤王响应。
- 主和压力。
- 禁军欠饷。
- 粮价指数。
- 派系积怨和恐惧。

检测异常：

```text
国库连续 6 回合单调上升且无代价 → 可能钱太多。
金军威压 3 回合内从 40 到 100 → 可能压迫过猛。
主和压力永远低于 20 → 主和派没有存在感。
民心长期高于 90 → 征发和粮价惩罚太轻。
某事项每局都在第 2 回合崩盘 → 玩家没有操作窗口。
```

## 数值护栏

### 硬护栏

硬护栏用于防止 LLM 或内容异常把数值打穿。

- 0-100 指标必须钳制。
- 钱粮不能无来源增加。
- 抄没收入必须绑定证据或案件。
- 调兵必须有军队、路线、粮草。
- 一道诏令单回合不得让核心指标变化超过上限，除非是结局事件。

### 软护栏

软护栏用于平衡体验。

| 行动 | 软上限 |
| --- | --- |
| 单次抄没入库 | 不超过当前国库 40% |
| 单回合城防提升 | 不超过 20 |
| 单回合金军威压下降 | 不超过 18 |
| 单回合民心下降 | 普通行动不超过 15 |
| 单回合派系恐惧上升 | 不超过 25 |

超过软上限可以发生，但必须有强叙事理由和强代价。

## 路线过强检测

### 清算路线过强

症状：

- 抄没持续解决财政。
- 派系反扑不构成威胁。
- 玩家无需粮运和借款。

修正：

- 提高证据门槛。
- 抄没后增加财税怠工。
- 商户和转运网络信用下降。
- 恐惧过高触发毁证和逃亡。

### 议和路线过强

症状：

- 假议和总能低代价拖延。
- 金军不惩罚失信。
- 主战清议没有反应。

修正：

- 增加金方识破概率。
- 失信后谈判条件恶化。
- 主战清议和京城士民反弹。
- 拖延必须消耗财物、质子或外交筹码。

### 守城路线过强

症状：

- 只修城就能赢。
- 禁军欠饷、粮价、勤王无关紧要。

修正：

- 城防提升受钱粮、民夫、器械限制。
- 守城战消耗粮和军心。
- 城门内应与火患独立威胁。

### 勤王路线过强

症状：

- 发诏后援军稳定抵达。
- 路线和补给没有压力。

修正：

- 勤王响应受君威、赏格、路线风险影响。
- 金军可分兵阻援。
- 空头赏格降低后续响应。

## 早崩检测

早崩不是难度高，而是玩家没有理解和介入窗口。

需要标记：

- 第 1-3 回合城破。
- 第 1-3 回合主和逼宫不可逆。
- 第 1-3 回合国库和粮储同时归零。
- 第一场对质必败且无解释。
- 首次金军事件直接导致结局。

修正原则：

- 前 3 回合给玩家至少一个成功爽点。
- 第一次失败只暴露系统，不直接锁死。
- 危机应先预警，再爆发。
- 教程事件应提供明确救火路径。

## 自动报告

每次跑局生成：

```text
runs/balance/YYYYMMDD-HHMM/
  summary.md
  metrics.csv
  endings.json
  route_compare.md
  anomaly_report.md
  sample_turns/
```

### summary.md

内容：

- 总局数。
- 结局分布。
- 平均回合数。
- 各路线胜率。
- 最常见失败原因。
- 异常种子列表。

### route_compare.md

按路线展示：

- 平均国库。
- 平均城防。
- 平均主和压力。
- 平均金军威压。
- 关键行动使用次数。
- 成功和失败的代表局。

### anomaly_report.md

自动列出：

- 数值飞天。
- 早崩。
- 无限循环。
- 诏令无效。
- 派系不反应。
- 某事件频率过高。

## 回归测试种子

固定一组“经典坏局”作为回归：

| 种子 | 用途 |
| --- | --- |
| `jingkang-standard-001` | 标准体验 |
| `treasury-crash-001` | 财政压力 |
| `grain-crisis-001` | 粮价压力 |
| `pro-peace-surge-001` | 主和压力 |
| `faction-backlash-001` | 清算反扑 |
| `jin-rush-001` | 金军高压 |
| `qinwang-delay-001` | 勤王迟缓 |
| `city-gate-risk-001` | 城门内应 |

每次大改都要跑这些种子，确保旧问题不回来。

## 人工复盘面板

自动跑局后，设计者需要能快速看一局：

- 每回合朝报。
- 自动玩家选择理由。
- 诏令列表。
- 月末回奏。
- 指标折线。
- 派系时钟。
- 失败前 3 回合发生了什么。

不要只给 CSV。平衡设计需要能读出故事。

## MVP 实施顺序

1. 先做固定脚本自动玩家，不接 LLM 决策。
2. 记录每回合核心指标和结局。
3. 生成 summary.md 和 anomaly_report.md。
4. 加 8 个路线策略。
5. 加派系时钟和诏令使用统计。
6. 加真实 LLM 自动玩家，用来找脑洞路线。

自动游玩不是替代人工设计，而是让设计师每天都能知道：这台靖康机器有没有变成无脑爽游，或者无情绞肉机。
